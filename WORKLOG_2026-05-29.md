## 1. CommentsSheet centralizado

**Arquivo criado:** `src/components/CommentsSheet.tsx`

**Decisões arquiteturais:**
- Base: `@gorhom/bottom-sheet` (NÃO `<Modal>` do RN — ver seção "Caminhos sem saída")
- `BottomSheetTextInput` para o input (integração nativa com a animação do sheet)
- `BottomSheetFlatList` pra lista, `BottomSheetFooter` pro input, `BottomSheetBackdrop` (opacity 0, sem sombra)
- Snap único `["85%"]` + `enableDynamicSizing={false}` (sem isso abre em ~5% pelo content)
- Adapter interno `adapter[type]` com endpoints diferentes pra `"post"` vs `"diet"`

**Props públicas:**
```ts
type CommentsTargetType = "post" | "diet";
interface CommentsSheetProps {
  type: CommentsTargetType;
  targetId: string | number | null;
  authorId?: string | number;       // habilita swipe-to-delete pro dono
  onCountChange?: (delta: number) => void;
}
interface CommentsSheetHandle { expand(): void; close(): void }
```

**Pattern de uso (importante):** o sheet é renderizado **uma vez no nível da screen hospedeira**, fora de qualquer `<ScrollView>`/`<FlatList>`. Cards (`PostCard`, `DietCard`) só emitem `onCommentPress(data)`. A screen mantém state `{ type, targetId, authorId }` e chama `commentsSheetRef.current?.expand()`.

**Por que lifted:** se cada card renderiza sua própria `<CommentsSheet>`, a árvore React fica `FlatList > Card > BottomSheet > BottomSheetFlatList` → dispara o warning `VirtualizedLists should never be nested inside plain ScrollViews`. Lifting elimina o warning.

## 3. AppDataContext loop fix

**Sintoma:** ~10 req/s em loop após login. Mesmo `/treinos`, `/dietas`, `/feed`, `/admin/*` se repetindo.

**Causa:** cadeia de deps instáveis no `useCallback`:
1. `fetchHomeData` dependia de `trainings.length`
2. Fetch completa → `setTrainings(mapped)` → length muda
3. Callback recriado → `refreshAll` (deps inclui ela) recriado
4. `useEffect` do auth (deps inclui `refreshAll`) re-executa
5. Chama `refreshAll(force=true)` que bypassava o TTL
6. API dispara → goto 2

**Fix em `src/contexts/AppDataContext.tsx`:**
- Removido `.length` de todos os fetch callbacks
- Trocado `dep: user` por `user?.sessionId, user?.isPendingSync` (e `user?.role` no admin)
- `if (X.length === 0) setLoadingX(true)` virou `setLoadingX(true)` direto
- `refreshAll` não passa mais `force=true` — TTL filtra bursts. Pull-to-refresh chama `fetchX(true)` direto.

## 5. UX final do CommentsSheet

- **Backdrop:** `opacity={0}` (mantém tap-fora-fecha, sem fundo escurecido)
- **Keyboard:** `keyboardBehavior="interactive"` + `keyboardBlurBehavior="restore"` + sem `android_keyboardInputMode` (default `adjustPan` evita double-lift que `adjustResize` causava)
- **Safe area Android:** `BottomSheetFooter` recebe `bottomInset={Platform.OS === "android" ? insets.bottom : 0}` pro input ficar acima da nav bar quando teclado fechado

## 🚫 Caminhos sem saída (não tentar de novo)

Pra teclado dentro de RN `<Modal>` (todos falharam):
1. `useReanimatedKeyboardAnimation` (react-native-keyboard-controller)
2. `useSharedValue` + `Keyboard.addListener` + `withTiming`
3. `useAnimatedKeyboard` (reanimated)
4. `KeyboardAvoidingView` do `react-native-keyboard-controller` (drop-in)
5. `Reanimated.View` com `paddingBottom` animado por SV

**Por quê:** RN `Modal` cria container nativo separado; eventos de teclado (especialmente `hide` no Android) não chegam confiavelmente nos listeners. Solução: abandonar `Modal`, usar `@gorhom/bottom-sheet` (vive na árvore React principal).
